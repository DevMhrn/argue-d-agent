# Lumen — Database Schema

This directory holds the SQL schema for the Supabase Postgres instance that all three pipeline stages share.

```
db/
└── migrations/
    ├── 001_initial.sql       schema for all tables, triggers, indexes
    └── 002_seed_statutes.sql public statute text (CA-1431.2, CVC-21453)
```

## How to apply

Paste each file, in order, into the Supabase SQL editor of a fresh project and run it. The schema is idempotent — re-running is safe.

> A more formal migration tool (Supabase CLI, sqitch, flyway) can be added later. For hackathon scope, paste-and-run is enough.

## What goes where

The schema implements a three-stage handoff. Each stage owns specific tables; downstream stages read upstream tables and flip status flags to trigger themselves.

| Stage | Owner | Writes | Reads | Triggered by |
|---|---|---|---|---|
| Ingestion | Aman | `documents`, `document_pages`, `cases.ingestion_complete=true` | (uploaded files) | new case POST |
| Ledger | Gowtham | `nodes`, `edges`, `cases.ledger_complete=true` | `documents`, `document_pages`, `statutes` | `cases.ingestion_complete=true` |
| Orchestration | Sudharsan | `transcript`, `decisions`, `cases.finalized=true` | `cases`, `nodes`, `edges`, `document_pages`, `statutes` | `cases.ledger_complete=true` |

## Tables at a glance

**`cases`** — one row per subrogation case. Carries the human-readable identifier, jurisdiction, headline damages, and three boolean flags that drive the cross-stage handoff. `tenant_id` has a default UUID so single-tenant demo writes "just work"; multi-tenant deployments override per request.

**`documents`** — one row per uploaded file, content-addressed by SHA-256. Raw bytes live in object storage (Backblaze B2); we keep `storage_bucket` + `storage_key` so signed URLs can be regenerated. `(case_id, sha256)` is unique → re-uploading the same file is idempotent. `document_kind` preserves the human-readable category ("police report", "FNOL") that the orchestration's `ClaimInput.documents[].kind` expects.

**`document_pages`** — one row per logical page of extracted text. PDFs use native pages; DOCX/HTML use heading or section boundaries; plain text is a single page. Granularity matters because Fact nodes reference `(document_id, page_number)` for the Fact Gate's verbatim-quote check. A GIN full-text index on `extracted_text` enables Postgres-native keyword search ("grep-equivalent" without embeddings).

**`statutes`** — the public legal text the Citation Gate validates statute citations against. `statute_id` is globally unique (e.g. `CA-1431.2`); jurisdiction-conflicting identifiers haven't appeared in practice and can be handled by a follow-up migration if they do.

**`nodes`** — the Evidence Ledger as a typed graph. `node_id` is the human-readable display id (`F1`, `P1`, `V1`...) unique within a case. `type` enumerates the supported node types (Fact, Party, Vehicle, Event, Location, Statute, Damage, Document); extend via `ALTER TABLE ... DROP CONSTRAINT / ADD CONSTRAINT`. For Fact nodes, `verbatim_quote + source_document_id + source_page_number` is the anchor the Fact Gate checks against `document_pages.extracted_text`. `source_document_id` uses `ON DELETE SET NULL` so historical facts survive their source documents being removed — audit-trail preservation.

**`edges`** — typed relationships between nodes. Edge types: `mentioned_in` (Fact → Document), `corroborates` / `contradicts` (Fact → Fact), `attributed_to` (Fact → Party), `governed_by` (Event → Statute), `caused` (Event → Damage), `involves` (Event → Party), `occurred_at` (Event → Location), `drives` (Party → Vehicle). Both directions are indexed so graph traversal is cheap either way.

**`transcript`** — Band-room postings per pipeline run. `run_id` is generated by the orchestrator at run start; `(run_id, seq)` is the canonical ordering. Same case can have many runs; UI shows the latest unless asked otherwise.

**`decisions`** — one row per pipeline run, holding the `FinalDecision` shape from `backend/app/types.py`. `secondary_decision` carries Adjudicator B's full output when dual-adjudication is on. `fault_table` is `jsonb` because its shape is structured but variable-length (`{factId, favors, weight}[]`); a `jsonb_typeof = 'array'` CHECK guards against the wrong root type at write time.

## Conventions

**UUIDs everywhere.** Primary keys are `gen_random_uuid()` — non-guessable, distributed-write-safe, and a stable reference across services. `node_id` and `edge_id` are separate human-readable strings (`F1`, `E1-2`) used for citations and display.

**Timestamps in UTC.** All `*_at` columns are `timestamptz`; Postgres stores UTC and converts to client TZ at display.

**CHECK constraints over Postgres ENUMs.** Enums are stricter but require `ALTER TYPE` to extend, which locks the type. CHECK constraints with allowed-value lists are easier to migrate.

**Cascade deletes for parent–child.** `documents → document_pages`, `cases → documents`, `cases → nodes`, `nodes → edges`, `cases → transcript`, `cases → decisions` — deleting a case cleans up everything it owns. The one exception: `nodes.source_document_id` uses `SET NULL`, preserving historical facts when their source document is removed.

**JSONB for variable-shape data.** `nodes.props`, `edges.props`, `decisions.escalate_reasons`, `decisions.fault_table`, `decisions.secondary_decision`, `document_pages.extraction_metadata`. JSONB is indexed and queryable; `json` is just text storage and slower.

## What is NOT here (intentional)

- **No embeddings, no `pgvector`, no `document_chunks`.** The ledger fits in agent prompts at our scale; semantic search is not needed for the demo. Full-text search via the GIN index covers the few keyword-lookup use cases. If we ever add embeddings, add a separate `document_chunks` table in a future migration.
- **No multi-tenant Row Level Security policies.** `tenant_id` columns are in place; RLS policies can be added later (`ALTER TABLE ... ENABLE ROW LEVEL SECURITY; CREATE POLICY ...`) without schema changes.
- **No `runs` table.** `run_id` is a bare UUID generated by the orchestrator. A `runs` table tracking start/end/status can be added later.
- **No vector embeddings on `nodes`.** The graph's typed edges substitute for similarity search at our scale.

## Quick sanity check after applying

```sql
-- Should return 8 rows: cases, documents, document_pages, statutes, nodes, edges, transcript, decisions
select table_name from information_schema.tables
where table_schema = 'public'
  and table_type = 'BASE TABLE'
order by table_name;

-- Should return 2 rows: CA-1431.2, CVC-21453
select statute_id, title from statutes order by statute_id;
```

## Open follow-ups (out of scope for migration 001)

- Multi-tenant Row Level Security policies.
- Soft-delete columns (`deleted_at`) if we ever want recoverable deletes.
- `runs` table tracking pipeline-execution metadata.
- Optional `document_chunks` table + `embedding vector(1536)` column if we adopt semantic search.
- A `cases_summary` view that pre-joins document counts and statuses for the frontend's case list.
